# PROJECT.md — how this repository works

Internal map of the `mcp-servers` project: what each file does, who calls whom,
and what it takes to run. For diagrams and design doctrine see
[architecture.md](architecture.md). For install/quick-start see
[README.md](README.md).

## What this is

Two [MCP (Model Context Protocol)](https://modelcontextprotocol.io) servers
that let an LLM agent operate a Cisco network lab end-to-end through
natural-language prompts:

- **cml-mcp** — the *lab control plane*: builds topologies on a Cisco Modeling
  Labs 2.8 controller, renders and applies day-0 bootstrap configs, manages
  lab/node lifecycle, wipes/deletes with confirmation gates. 23 tools.
- **network-device-mcp** — the *device data plane*: SSH into booted IOS-XE /
  NX-OS devices for show commands, interface/OSPF/BGP/ACL configuration, and
  parsed neighbor verification. 13 tools.

They are two servers (not one) because they sit on different **credential and
API boundaries**: CML admin JWT vs device SSH credentials. One server per
auth boundary is a project rule.

## Who calls whom

```
You (prompt)
  └─▶ MCP HOST (Claude Code today; own local host / hiTech Automation AI planned)
        ├─▶ LLM brain (Anthropic API today; Ollama planned) — decides tool + args
        ├─▶ MCP CLIENT #1 ── stdio JSON-RPC ──▶ cml_mcp_server.py
        │                                          └─▶ virl2_client ── HTTPS /api/v0 (JWT) ──▶ CML controller ──▶ lab VMs
        └─▶ MCP CLIENT #2 ── stdio JSON-RPC ──▶ device_mcp_server.py
                                                   └─▶ platforms/<os>.py (render/parse)
                                                         └─▶ Netmiko ── SSH :22 ──▶ devices
```

Terminology: the **host** is the application (owns the LLM loop, approval
gates, config); an **MCP client** is the 1:1 connection object inside the host,
one per server; a **server** provides tools. The host spawns each server as a
child process and talks JSON-RPC 2.0 over its stdin/stdout
(`initialize → tools/list → tools/call`). Tool schemas are generated by
fastmcp from Python type hints and docstrings — docstrings are the
agent-facing API.

Inside each server, tools never call other tools; shared logic lives in
private `_impl` helpers (`_connect`, `_send_config_impl`, `_run_show_impl`,
lazy `cml()` singleton).

## Call flow for one operation (example)

`"Enable OSPF 1 area 0 on router1, then verify"`:

1. Host sends prompt + all tool schemas to the brain
2. Brain returns `tool_use: configure_ospf(...)` → host shows **approval gate**
3. Host routes `tools/call` to the netdev client → server
4. Server: `iosxe.render_ospf()` → command list → Netmiko push → echo scanned
   into `rejected_lines` → structured result back
5. Brain chains `check_ospf_neighbors` → server runs the show, regex-parses,
   returns `{parsed, full_count, raw}` — raw included so results can be
   reconciled field-by-field
6. Brain writes the answer; host prints it

## Software requirements

| Component | Version / note |
|---|---|
| Python | 3.11+ (project developed on 3.13) |
| [uv](https://docs.astral.sh/uv/) | environment + dependency manager (replaces venv/pip) |
| fastmcp | MCP server + client library (3.x) |
| virl2-client | official CML Python client — **match your controller version** (2.8.x here) |
| netmiko | SSH transport for device tools |
| python-dotenv | `.env` loading (each server loads its own, path-anchored) |
| jinja2 | day-0 template rendering (cml-mcp only) |
| Cisco CML | 2.8 controller, reachable over HTTPS |
| MCP host | Claude Code (tested); any stdio-capable MCP host works |

Infrastructure this was built against: CML 2.8 VM, cat8000v (IOS-XE 17.x) and
nxosv9000 (9.3) nodes, all on a bridged 192.168.89.0/24 lab network. The
device server also drives standalone (non-CML) VMware-hosted devices — any
SSH-reachable IOS-XE/NX-OS box works.

## Configuration

Each server reads a `.env` in its own directory (see each `.env.example`):

- `cml-mcp/.env`: `CML_URL`, `CML_USER`, `CML_PASS` (controller, JWT auth) +
  `DAY0_USER`, `DAY0_PASS`, `DAY0_PASS_HASH_NXOS` (rendered into day-0
  configs; the NX-OS type-5 hash exists because NX-OS mis-ingests cleartext
  passwords during day-0 config replay)
- `network-device-mcp/.env`: `DEVICE_USER`, `DEVICE_PASS` (one pair for all
  devices)

`.env` files are chmod 600 and git-ignored. Note: the MCP boundary is a
convenience boundary, not a security boundary — an agent with shell access on
the same machine can read these files; real isolation needs an OS boundary.

## Operational rules of thumb

- After editing any server file, **reconnect the server in the host** — the
  running child process never picks up file edits.
- Day-0 config changes only take effect when a node boots from wiped
  (`DEFINED_ON_CORE`) state.
- New adjacencies need time (OSPF ~40 s to FULL; NX-OS boots 5–8 min).
- Destructive tools (`wipe_node`, `delete_lab`) require `confirm=true` *and*
  should stay behind the host's approval prompts.
